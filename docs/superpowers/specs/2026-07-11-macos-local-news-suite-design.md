# macOS 本地公众号资讯套件设计

日期：2026-07-11

状态：已批准进入书面规格复核

适用分支：`feat/collection-pilot`

## 1. 背景

现有 `inno-portfolio-collector` 已验证一条面向 10 个既定项目的本地采集与交付链路：通过 `MooreExporterAdapter` 调用 Moore 下载器，完成精确公众号映射、2026 日期过滤、增量去重、失败隔离、正文与附件校验、Obsidian Vault 生成及安全 ZIP 交付。当前基线为 225 篇文章、642 个附件、10 个项目、0 断链、0 哈希错误、0 敏感信息和 239 项测试通过。

下一阶段不是继续扩充一次性脚本，而是把已经验证的能力包装成无需 Codex、可安装的 macOS 本地桌面产品。产品吸收 `wechat-article-exporter` 的搜索、目录、正文获取和管理界面经验，保留 Moore 下载器的本地命令封装，并继续以本项目的业务规则与交付校验作为可信核心。

## 2. 已确认的产品决策

1. 首版采用集中采集：只有资料维护者在自己的 Mac 上登录和采集，朋友端不包含登录、扫码或采集能力。
2. 一个代码库构建两个 macOS 应用：采集端和朋友端。
3. 采集正文与附件为只读来源；朋友在独立区域编写笔记、摘要、选题稿和编辑稿。
4. 资料维护者通过带版本号的增量更新包向朋友分发新内容。
5. 朋友通过独立编辑稿包回传人工成果；AirDrop、微信或其他传输渠道均可，产品不绑定渠道。
6. 首版无云端服务，全部核心功能离线可用。
7. Obsidian 是推荐的深度阅读与编辑工具；离线 HTML 是只读结果看板，不是第二套数据源。
8. 首版不包含 DeepSeek-V4-Pro 对话、自动发布或复杂执行动作，但保留未来接入边界。

## 3. 目标与非目标

### 3.1 目标

- 让资料维护者通过图形界面完成预检、采集、状态查看、校验和增量包生成。
- 让朋友通过图形界面安全导入更新、搜索与阅读文章、查看离线看板、管理独立编辑稿，并一键打开 Obsidian Vault。
- 保证来源内容更新不会覆盖人工编辑稿。
- 保证任一项目采集失败不会阻断其他项目，也不会把不完整结果显示为完整成功。
- 保持所有数据为可检查的本地文件，用户可在脱离应用后继续读取 Markdown、附件和 HTML。
- 让现有 239 项测试覆盖的采集与安全逻辑继续作为回归基线，而不是重写后丢失验证成果。

### 3.2 非目标

- 不让朋友端自行采集公众号或持有任何采集登录状态。
- 不提供多人实时协同、云同步、账号系统、权限后台或自动冲突合并。
- 不修改用户原始项目清单，也不在首版提供模糊公众号匹配。
- 不自动改写采集正文，不代替人工判断文章版权、选题价值或发布授权。
- 不承诺抓取付费、私密、未公开或页面未暴露的数据。

## 4. 产品组成

### 4.1 采集端应用

采集端仅分发给资料维护者，包含以下板块：

- **概览**：显示项目成功度、最近采集版本、文章与附件数量、失败项目和校验状态。
- **采集**：执行登录状态检查、精确账号核验、目录同步、日期过滤和增量下载。
- **资料库**：按项目、公众号、日期和状态浏览只读来源，并预览即将分发的变化。
- **交付**：生成完整基线包或增量更新包，展示包版本、变化摘要、哈希和失败信息。
- **稿件收件箱**：导入朋友回传的编辑稿包；同名或同版本冲突不得静默覆盖。

采集端继续通过窄接口调用采集后端。首版默认后端是现有 Moore 适配器；界面不得直接依赖 Moore 的内部文件结构。未来若接入 `wechat-article-exporter` 的另一种本地后端，只需实现同一组认证检查、账号列表、目录同步、文章列表和下载接口。

### 4.2 朋友端应用

朋友端不包含采集后端、扫码入口或凭据目录访问能力，包含以下板块：

- **阅读**：搜索、筛选、打开文章与附件，显示来源、发布日期、项目和采集版本。
- **看板**：打开随资料包生成的完全离线 HTML 结果看板。
- **编辑**：从文章创建独立笔记、摘要、选题稿或编辑稿；来源文章只可引用，不可覆盖。
- **更新**：导入基线包或增量更新包，预览变化并在校验通过后原子应用。
- **导出**：把选中的人工稿件及其来源引用导出为编辑稿包。
- **Obsidian**：打开当前 Vault；未安装 Obsidian 时给出安装说明，不阻断应用内阅读。

## 5. 架构

采用 **Vault-first 双应用 + 共享核心**：

```text
公众号公开内容
      │
      ▼
采集后端接口 ── Moore 适配器（首版默认）
      │
      ▼
共享内容核心 ── 映射 / 过滤 / 去重 / 校验 / Vault / 包协议 / 看板
      │                              │
      ▼                              ▼
采集端应用                    版本化更新包
                                     │
                                     ▼
                              朋友端应用
                                     │
                         ┌───────────┴───────────┐
                         ▼                       ▼
                    本地 Vault               编辑稿包
                         │                       │
                         ▼                       ▼
                     Obsidian                采集端收件箱
```

两个应用使用原生 macOS 交互壳。现有 Python 业务核心按职责拆成可独立打包的本地助手：采集能力只进入采集端；包校验、Vault 更新和看板生成等无凭据能力可由两端复用。应用与本地助手只通过版本化 JSON 请求/响应通信，不解析对方日志或依赖终端输出。应用安装包必须自带所需运行环境，朋友无需安装 Python、Codex 或上游仓库。

## 6. 本地资料库

Vault 是唯一可信内容库，离线 HTML 和应用索引都可由 Vault 重建。首版沿用现有来源目录，并新增明确隔离的人工与看板区域：

```text
英诺被投项目资讯库/
├── 00-首页.md
├── 01-采集状态.md
├── 02-项目/              # 只读项目索引
├── 03-文章/              # 只读采集正文
├── 04-附件/              # 只读采集附件
├── 10-编辑稿/            # 人工可编辑成果
├── 11-个人笔记/          # 人工可编辑阅读笔记
├── 80-离线看板/          # 可重建的静态 HTML 与本地资源
└── 90-系统/              # 清单、报告、版本和迁移信息
```

更新包只能写入只读来源、离线看板和系统元数据区域；不得写入或删除 `10-编辑稿`、`11-个人笔记`。离线看板由采集端生成并随资料包分发，朋友端只在看板缺失或格式迁移时从同一份系统清单重建。编辑稿包只能写入采集端的暂存收件箱，经用户确认后进入人工区域，不得改动来源区域。

## 7. 包协议

### 7.1 基线包与增量更新包

每个资料包包含版本化清单、变化文件、内容哈希、来源版本、目标版本、创建时间、项目统计和采集报告。增量包必须声明它所基于的资料库版本；朋友端只在本地版本匹配且全部校验通过时应用。

安全规则沿用并扩展现有 ZIP 校验：拒绝绝对路径、路径穿越、符号链接、特殊文件、大小超限、macOS 大小写或 Unicode 路径碰撞、未声明文件、哈希不符、敏感凭据痕迹和不兼容协议版本。导入先解压到隔离暂存区，再校验，再原子替换来源区域；任何失败都保持原资料库不变。

首版的哈希保证内容完整性，但不建立账号或证书体系。朋友只应导入由资料维护者通过已确认渠道发送的包；应用在导入确认页显示创建者标签、版本和摘要，不把未知来源包描述为可信。

### 7.2 编辑稿包

编辑稿包仅包含人工稿件、必要的本地附件、稳定稿件 ID、作者标签、导出时间、稿件版本和所引用来源文章的稳定 ID。它不包含采集登录信息、整个资料库或未引用的来源正文。

采集端导入时按稳定稿件 ID 和版本判断：新稿直接进入收件箱；已存在但内容不同的稿件以并列版本呈现，必须由用户选择保留或合并，禁止“最后写入者覆盖”。

## 8. 数据流

### 8.1 采集与交付

1. 采集端检查本地登录状态和 10 个项目的精确账号映射。
2. 用户先查看预检结果，再显式开始采集。
3. 各项目独立同步与下载；部分失败进入报告，成功项目继续。
4. 共享核心过滤日期、去重、校验正文与附件，并生成候选版本。
5. 用户查看变化摘要和失败信息；只有校验通过的候选版本才能生成包。
6. 包生成后可通过任意外部渠道发送，应用不接管传输账号。

### 8.2 朋友更新与编辑

1. 朋友端读取包清单并显示新增、变更、失败项目和目标版本。
2. 校验通过后原子更新只读来源区与包内看板，并重建应用索引；看板缺失或格式迁移时才从系统清单重建。
3. 朋友从来源文章创建独立稿件；稿件只保存来源稳定 ID，不复制可被更新覆盖的路径假设。
4. 朋友选择稿件导出编辑稿包，再通过任意渠道回传。

## 9. 离线 HTML 看板

离线看板从系统清单和采集报告生成，不读取外部接口，不引用 CDN、远程字体或远程脚本。它至少展示项目覆盖率、成功/部分成功状态、文章与附件数量、时间分布、最近更新、失败摘要，并提供本地标题与项目筛选。

看板是可重建的只读衍生物。看板生成失败不得损坏 Vault 或阻止用户在应用、Finder、Markdown 或 Obsidian 中阅读；应用应明确标记“资料已更新、看板生成失败”。

## 10. 错误处理

- 登录失效、账号不唯一或上游不可用：只阻止受影响的采集动作，不删除旧版本。
- 单项目同步或下载失败：记录为部分成功，其他项目继续；交付摘要必须显眼显示不完整范围。
- 包版本不匹配：拒绝应用并保留当前资料库，提示需要基线包或缺失的中间版本。
- 包内容损坏或路径不安全：隔离并拒绝，不向正式 Vault 写入任何文件。
- 索引或看板重建失败：保留已校验的来源与人工稿件，允许稍后重试衍生物生成。
- Obsidian 不存在或打不开：应用内功能继续可用，仅提供安装或重新定位 Vault 的提示。
- 编辑稿冲突：创建并列版本并要求人工处理，不自动合并或覆盖。

所有面向用户的诊断都必须脱敏；日志不得记录 Cookie、Token、auth-key、pass_ticket 或完整凭据目录内容。

## 11. 合规与署名

- 两个上游项目均按 MIT 许可证要求保留许可证文本、版权声明和适用的源码修改说明。
- 应用的“关于”页面和分发包包含第三方许可清单；如果后续复制或修改上游前端源码，必须明确标出来源与修改范围。
- 首版不向朋友分发采集端、上游运行目录、Cookie、Token 或任何登录凭据。
- 文章版权归原作者或权利人；产品界面和导出内容保留来源链接、公众号名称和发布日期，并提示仅在有权范围内阅读、编辑和转发。
- 公众号运营人员的身份不自动等同于采集授权；采集行为仍由资料维护者本人负责。

## 12. 测试与验收

### 12.1 自动化测试

- 保持现有 239 项测试全部通过，并把它们作为采集核心回归门槛。
- 为基线包、增量包和编辑稿包建立固定夹具与跨版本兼容测试。
- 覆盖重复导入、乱序导入、版本缺口、哈希错误、路径穿越、符号链接、路径碰撞、大小限制和敏感信息检测。
- 覆盖来源区不可编辑、更新不触碰人工区、编辑稿冲突不覆盖的契约测试。
- 覆盖离线看板无外部网络依赖及从 Vault 完整重建。
- 覆盖应用与本地助手的 JSON 协议版本、超时、取消和异常输出。

### 12.2 端到端验收

1. 在一台干净 Mac 上安装采集端，无需 Codex 即可完成预检、一次增量采集和包生成。
2. 在另一台干净 Mac 上安装朋友端，无需 Python 即可导入基线与后续增量包。
3. 导入后能离线搜索和阅读 10 个项目资料，打开 HTML 看板，并可用 Obsidian 打开 Vault。
4. 创建人工稿件后再次导入更新，稿件内容和来源引用保持不变。
5. 编辑稿包能被采集端导入；冲突版本不会静默覆盖。
6. 任一损坏或恶意构造的包都不能部分写入正式 Vault。
7. 分发物中不包含敏感信息，并包含两个 MIT 上游的适用许可证与署名。

## 13. 未来扩展边界

未来的文章问答功能应作为读取本地资料库的独立能力接入。它只能访问用户明确选择的来源与稿件，回答必须保留文章引用，不得直接修改来源正文；模型密钥、费用上限、缓存和联网提示需要另行设计。DeepSeek-V4-Pro 对话不属于本规格的首版范围。

## 14. 首版完成定义

首版完成必须同时满足：双应用可独立安装；采集登录态只存在于采集者 Mac；版本化增量更新不会覆盖人工区；编辑稿包完成回传闭环；离线 HTML 与 Obsidian 均可使用；既有采集与安全回归通过；分发包完成敏感信息、许可证和署名检查。
